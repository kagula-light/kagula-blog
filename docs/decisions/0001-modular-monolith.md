# ADR-0001：采用 Next.js 模块化单体

状态：已接受
日期：2026-07-10

## 背景

项目需要公开博客、管理后台、用户系统、互动和热点采集。目标服务器只有约 2 GB 内存，并已有其他服务。用户希望拥有自己的后台和 CI/CD。

## 决策

采用 pnpm workspace：

- 一个 Next.js Web 应用承载页面、后台和浏览器 API。
- 一个同仓库 Worker 处理热点与定时任务。
- PostgreSQL、Redis 和 R2 提供数据、缓存和文件能力。
- 使用功能模块保持内部边界。

## 结果

优点：

- 部署和类型共享简单。
- SEO 与服务端渲染直接。
- 资源开销低于独立 API、Web 和多个微服务。
- Worker 失败可与 Web 隔离。

代价：

- 必须严格维护模块依赖，避免所有业务堆入 Next.js 路由。
- Web 与 Worker 共享数据库，需要清晰事务与幂等规则。
- 将来拆分服务时需要稳定契约。

## 未选择

- Next.js + 独立 NestJS/Go API：V1 运维和资源成本过高。
- 全托管后端：增加外部依赖，不符合自托管目标。
- 微服务：当前没有团队或规模信号支持。
